## 作るもの: MyAnalytics

MBTI などの**診断系アプリの結果を登録して、グラフィカルに表示・共有できる**アプリです。

主な機能は次の 4 つです。

1. **認証** — メールアドレスとパスワードでサインアップ / ログイン
2. **結果の登録** — 16Personalities などの結果 URL を貼ると、URL から「何の診断か・結果が何タイプか」を判定して登録。判定できない場合は手動選択にフォールバック
3. **グラフィカル表示** — 登録した結果を **5 種類のチャート**(レーダー / 棒 / ドーナツ / カードグリッド / スライダー)× **5 種類のテーマ**(ライト / ダーク / ポップ / ミニマル / サイバー)で表示
4. **共有** — 発行された共有 URL を送ると、ログインしていない人でも結果を見られる

::: tip なぜこの題材?
「認証」「外部データの取り込み」「マスターデータ設計」「グラフ描画」「公開ページ」という、業務アプリで頻出する要素が一通り入っているからです。作り終わるころには Rails アプリの典型的な構成要素を一巡できます。
:::

### 診断結果データの持ち方(重要な設計判断)

外部サイトの結果ページをスクレイピングして中身を取得することは**しません**(利用規約・JS 描画の問題で不安定なため)。かわりに:

- MBTI の 16 タイプの名称・説明文・特性は**マスターデータとしてアプリ内の DB に持つ**
- ユーザーが貼った URL からは「タイプ(INTP など)」だけを判定する
- 画面に表示する内容は、判定したタイプに対応する**アプリ内のマスターデータ**から出す

この「外部に依存しすぎない設計」も業務でよく使う考え方です。

## 全体アーキテクチャ

### 開発環境(第2〜10章)

`docker compose` で 3 つのコンテナを動かします。

```
┌─────────────────────── docker compose ───────────────────────┐
│                                                              │
│  frontend (Vite dev server)      api (Rails 8 API)           │
│  http://localhost:5173    ──────▶ http://api:3000            │
│  React + TypeScript        proxy   │                         │
│                                    │ SQL                     │
│                                    ▼                         │
│                                  db (PostgreSQL 16)          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- ブラウザは Vite の開発サーバー(5173 番)だけを見る
- `/api` へのリクエストは Vite の **proxy 機能**で Rails に転送する(CORS 問題を回避)
- Rails は JSON を返すことに専念する(API モード)

### 本番環境(第11章)

無料枠に収めるため、**React のビルド成果物を Rails の `public/` に同梱して 1 コンテナ**にします。

```
ブラウザ ──▶ Render Web Service(無料)──▶ Neon PostgreSQL(無料)
             Rails 8 コンテナ
             ・/api/* → Rails が JSON を返す
             ・それ以外 → public/ の React ビルドを返す
```

業務の ECS Fargate 構成との対応関係は[付録A](/chapters/appendix-a-ecs-fargate)で詳しく扱います。

## データモデルの全体像

先にゴールのテーブル構成を見ておきます。各テーブルは登場する章で詳しく設計します。

```
users ────┐
          │ 1:N
sessions ─┘(第4章: 認証)

diagnoses(診断マスター: まずは MBTI の 1 レコード)
   │ 1:N
diagnosis_types(タイプマスター: MBTI の 16 タイプ + 説明文)  (第3章)
   │ 1:N
results(ユーザーが登録した結果)                             (第6章)
   ├─ user_id / diagnosis_type_id
   ├─ source_url     … 貼り付けた結果 URL
   ├─ scores (jsonb) … 5 指標のスコア%
   ├─ share_token    … 共有 URL 用トークン(第9章)
   └─ chart_type / theme … 表示設定(第7・8章)
```

「MBTI 専用テーブル」にせず `diagnoses` → `diagnosis_types` の 2 段のマスターにしてあるのは、将来エニアグラムなど**別の診断を追加してもテーブルを増やさずに済む**ようにするためです。
